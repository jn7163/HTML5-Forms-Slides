Advantages

  • Makes form validation native to the browser.*
  • User Interface enhancements
  • Less JavaScript! More speedy pages!

(*Still need to validate on the server.)

Disadvantages (for now)

  • Not well-supported by currently-used browsers
    (Opera leads the pack!* Still need JavaScript.)
  • Little to no control over native UI in most browsers.
  • Pattern validation is not consistent across browsers.

*In truth, Chrome 10 does for now.

Demo 3: U.S. Telephone Numbers

	<input type="tel" name="telnum" value="" 
	placeholder="818-555-1212"
	pattern="^\(?([0-9]{3})\)?[-. ]?([0-9]{3})[-. ]?([0-9]{4})$"
	required>

See it in action (works best in Chrome 10)

Demo includes CSS 3 pseudo-classes and an optional field.

Demo 4: URLs with a pattern

<input type="url" name="urlfield" value="" 
placeholder="ftp://"
pattern="^(ftp(?:s)?\:\/\/[a-zA-Z0-9\-]+(?:\.[a-zA-Z0-9\-]+)*\.[a-zA-Z]{2,6}(?:\/?|(?:\/[\w\-]+)*)(?:\/?|\/\w+\.[a-zA-Z]{2,4}(?:\?[\w]+\=[\w\-]+)?)?(?:\&[\w]+\=[\w\-]+)*)$""
required>

See it in action

Demo includes CSS 3 pseudo-classes and an optional field.

Fall back strategies

Keep doing what we have been:


  • Validate data on the server
  • Use JavaScript based-error checking
  • Use existing User Interface libraries such as jQuery UI to mimic more complex elements

Remember: These attributes are stll a part of the DOM tree. Input fields default to type="text" in browsers that don't support them.
